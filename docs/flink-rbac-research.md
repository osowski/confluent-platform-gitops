# Flink RBAC Research: Implementation Guide

**Date:** March 17, 2026
**Objective:** Implement multi-user RBAC for Confluent Platform Flink to demonstrate user isolation at kubectl, Confluent CLI, and Control Center UI levels

---

## Executive Summary

### Goal
Demonstrate Flink RBAC with the following scenario:
- **Shapes FlinkApplications** in **shapes-env** (namespace: flink-shapes) → Only visible to **shapes group** (5 users)
- **Colors FlinkApplications** in **colors-env** (namespace: flink-colors) → Only visible to **colors group** (5 users)
- **All FlinkApplications** → Visible to **admin**

### User Groups
- **Shapes Group** (5 users): user-square, user-circle, user-triangle, user-trapezoid, user-diamond
- **Colors Group** (5 users): user-red, user-green, user-orange, user-blue, user-yellow
- **Admin**: admin (full cluster access)

### Access Control Levels
1. **Kubernetes kubectl** - Namespace-based RBAC with group-level permissions
2. **Confluent Flink CLI** - OAuth token-based authentication with group membership
3. **Control Center UI** - OIDC SSO with group-based role filtering

### Recommended Approach: **Hybrid Authentication**

**Why Hybrid?**
- ✅ Fastest setup (30-60 minutes)
- ✅ Best demo experience
- ✅ Production-relevant for UI access
- ✅ No complex Kind OIDC configuration needed
- ✅ Clear separation of concerns

**Components:**
1. **kubectl access** → Kubernetes RBAC with ServiceAccount-based kubeconfig contexts
2. **C3/CMF UI access** → OIDC with Keycloak (runs inside Kind cluster)
3. **Confluent CLI** → OAuth tokens from Keycloak

### kubectl Access Patterns

**Two Separate User Contexts:**

1. **Kind Cluster Admin** (Kubernetes platform administrator)
   - Access: Default kubeconfig from `kind get kubeconfig --name flink-demo-rbac`
   - Authentication: Client certificate (generated by Kind)
   - Authorization: `cluster-admin` ClusterRoleBinding (full cluster access)
   - **No Keycloak interaction** - standard Kubernetes admin access
   - Use case: Cluster management, debugging, emergency operations

2. **Demo Users (@osow.ski)** (Application developers/operators)
   - Access: ServiceAccount-based kubeconfig contexts (generated by setup script)
   - Authentication: ServiceAccount tokens (long-lived JWTs)
   - Authorization: Namespace-scoped Roles via RoleBindings (limited to assigned namespaces)
   - **Keycloak authentication required** for Control Center UI and CMF CLI access
   - Use case: Managing Flink applications within assigned environments

**Keycloak Deployment:**
- Primary approach: Runs inside Kubernetes cluster (StatefulSet with PostgreSQL)
- Alternative approach: External Docker Compose (documented in `clusters/flink-demo-rbac/README.md`)

1. **S3 credentials in ConfigMap** (`/workloads/cp-flink-sql-sandbox/base/cmf-config-configmap.yaml:70-71`)

   - Should use Kubernetes Secrets

2. **No Control Center authentication** - Anyone can access C3 UI

3. **No CMF authentication** - No user audit trail

4. **No encryption in transit** - HTTP-only between components

### Namespace Strategy

Create separate namespaces for logical isolation:

```yaml
# New namespaces to create
- flink-app1      # For FlinkEnvironment env1 and FlinkApplication1
- flink-app2      # For FlinkEnvironment env2 and FlinkApplication2
- keycloak        # For Keycloak deployment (optional: can run in Docker outside cluster)
```

**Rationale:**
- Kubernetes RBAC can restrict user access per namespace
- Each user can only see FlinkApplications in their assigned namespace
- Admin has cluster-wide access via ClusterRole
- FlinkEnvironments are namespace-scoped, enforcing natural boundaries

### 1.5 Generate kubeconfig Contexts

**Script:** `scripts/setup-rbac-kubeconfigs.sh`

```bash
#!/bin/bash
set -e

CLUSTER_NAME="flink-demo"
KUBECONFIG_DIR="$HOME/.kube/flink-rbac"
mkdir -p "$KUBECONFIG_DIR"

# Get Kind cluster endpoint
CLUSTER_ENDPOINT=$(kubectl config view -o jsonpath="{.clusters[?(@.name==\"kind-${CLUSTER_NAME}\")].cluster.server}")
CLUSTER_CA=$(kubectl config view --raw -o jsonpath="{.clusters[?(@.name==\"kind-${CLUSTER_NAME}\")].cluster.certificate-authority-data}")

# Function to create kubeconfig for a ServiceAccount
create_kubeconfig() {
    local SA_NAME=$1
    local SA_NAMESPACE=$2
    local CONTEXT_NAME=$3
    local OUTPUT_FILE="$KUBECONFIG_DIR/${CONTEXT_NAME}.kubeconfig"

    echo "Creating kubeconfig for ${SA_NAME}..."

    # Get ServiceAccount token (K8s 1.24+)
    SA_TOKEN=$(kubectl create token ${SA_NAME} -n ${SA_NAMESPACE} --duration=8760h)

    # Create kubeconfig
    cat > "$OUTPUT_FILE" <<EOF
apiVersion: v1
kind: Config
clusters:
  - cluster:
      certificate-authority-data: ${CLUSTER_CA}
      server: ${CLUSTER_ENDPOINT}
    name: ${CLUSTER_NAME}
contexts:
  - context:
      cluster: ${CLUSTER_NAME}
      namespace: ${SA_NAMESPACE}
      user: ${SA_NAME}
    name: ${CONTEXT_NAME}
current-context: ${CONTEXT_NAME}
users:
  - name: ${SA_NAME}
    user:
      token: ${SA_TOKEN}
EOF

    echo "✅ Created ${OUTPUT_FILE}"
}

# Create kubeconfigs
create_kubeconfig "user1" "flink-app1" "user1@flink-demo"
create_kubeconfig "user2" "flink-app2" "user2@flink-demo"
create_kubeconfig "admin" "default" "admin@flink-demo"

echo ""
echo "===== Kubeconfig Setup Complete ====="
echo "Use these commands to switch contexts:"
echo ""
echo "  export KUBECONFIG=$KUBECONFIG_DIR/user1@flink-demo.kubeconfig"
echo "  kubectl get flinkapplications"
echo ""
echo "  export KUBECONFIG=$KUBECONFIG_DIR/user2@flink-demo.kubeconfig"
echo "  kubectl get flinkapplications"
echo ""
echo "  export KUBECONFIG=$KUBECONFIG_DIR/admin@flink-demo.kubeconfig"
echo "  kubectl get flinkapplications --all-namespaces"
```

      # Expected token issuer (external URL for browser redirects)
      oauthbearer.expected.issuer: http://keycloak.flink-demo-rbac.confluentdemo.local:30080/realms/confluent

      oauthbearer.sub.claim.name: email

      # For service accounts (client_credentials), use client_id
      oauthbearer.fallback.sub.claim.name: client_id

      # Token endpoint for client credentials flow (internal cluster DNS)
      oauthbearer.token.endpoint.url: http://keycloak.keycloak.svc.cluster.local:8080/realms/confluent/protocol/openid-connect/token

      # JWKS refresh interval
      oauthbearer.jwks.refresh.ms: 3600000

  # Environment variables for OAuth
  envVars:
    - name: CMF_OAUTH_ENABLED
      value: "true"
    - name: CMF_OAUTH_PROVIDER
      value: "keycloak"

  # Mount OAuth JAAS configuration
  mountedSecrets:
    - secretRef: cmf-oauth-jaas
      mountPath: /mnt/secrets/oauth

# Resource limits
resources:
  requests:
    cpu: 2000m
    memory: 1Gi
  limits:
    cpu: 2000m

      # Token endpoint (internal cluster DNS for server-side calls)

      - confluent.controlcenter.auth.oidc.jwks.url=http://keycloak.keycloak.svc.cluster.local:8080/realms/confluent/protocol/openid-connect/certs

      # User info endpoint (internal cluster DNS)

      # Redirect URI (C3 callback URL)
      - confluent.controlcenter.auth.oidc.redirect.uri=https://controlcenter.flink-demo.confluentdemo.local/login/callback

      # Scopes and claims
      - confluent.controlcenter.auth.oidc.scope=openid profile email

## Security Considerations

### What This Implementation Provides

✅ **Kubernetes-level RBAC** - Users isolated via namespace permissions
✅ **UI-level authentication** - Keycloak OIDC SSO for Control Center
✅ **CMF authorization** - ConfluentRoleBindings enforce environment access
✅ **Audit trail** - User actions logged with email identifiers
✅ **Token-based access** - Short-lived OAuth tokens (5-minute default)

### What This Implementation Does NOT Provide

❌ **Kafka encryption** - Still using PLAINTEXT (no mTLS or SASL)
❌ **Schema Registry auth** - HTTP without authentication
❌ **Network policies** - No pod-to-pod traffic restrictions
❌ **Secrets encryption at rest** - Kubernetes Secrets not encrypted
❌ **Certificate rotation** - Static client secrets in Keycloak

### Production Enhancements Needed

1. **Enable Kafka mTLS/SASL:**
   - Configure Kafka listeners with SASL_SSL
   - Use cert-manager to issue client certificates
   - Update FlinkApplications to use authenticated connections

2. **Secure Schema Registry:**
   - Enable HTTPS with TLS certificates
   - Add RBAC integration with MDS
   - Update Flink SQL connectors with auth config

3. **Implement Network Policies:**
   - Restrict CMF to only talk to CFK API
   - Limit Flink pods to only Kafka/Schema Registry
   - Block external access to Keycloak from within cluster

4. **Encrypt Secrets:**
   - Enable Kubernetes encryption at rest
   - Use HashiCorp Vault for secret management (already deployed!)
   - Rotate OAuth client secrets regularly

5. **Add TLS for CMF:**
   - Enable HTTPS on CMF service
   - Update CMFRestClass with TLS configuration
   - Use cert-manager to issue certificates

6. **Harden Keycloak:**
   - Use PostgreSQL with persistent storage (already in docker-compose)
   - Enable HTTPS with valid certificates
   - Configure short-lived refresh tokens
   - Add MFA for admin users

## Troubleshooting

### kubectl Access Denied

**Problem:** User can't access resources in their namespace

**Solution:**
```bash
# Verify ServiceAccount exists
kubectl get sa user1 -n flink-app1

# Verify RoleBinding
kubectl get rolebinding user1-flink-developer -n flink-app1

# Check permissions
kubectl auth can-i get flinkapplications --as=system:serviceaccount:flink-app1:user1 -n flink-app1
```

### Control Center OIDC Redirect Loop

**Problem:** C3 keeps redirecting to Keycloak

**Solution:**
1. Verify Keycloak client redirect URIs include C3 URL
2. Check Control Center logs: `kubectl logs -n kafka controlcenter-0`
3. Ensure `confluent.controlcenter.auth.oidc.redirect.uri` matches actual C3 hostname
4. Verify Keycloak realm is `confluent` (not `sso_test`)

### CMF Cannot Authenticate to Keycloak

**Problem:** CMF fails to validate OAuth tokens

**Solution:**
```bash
# Test Keycloak JWKS endpoint from within cluster
kubectl run curl-test --image=curlimages/curl -it --rm -- \
  curl http://host.docker.internal:8080/realms/confluent/protocol/openid-connect/certs

# If fails, check:
# 1. Kind cluster has host-gateway extra mapping
# 2. Keycloak is running: docker ps | grep keycloak
# 3. Firewall not blocking port 8080
```

**Fix for host.docker.internal in Kind:**

Edit Kind cluster config (`clusters/flink-demo/kind-config.yaml`):

```yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
    extraPortMappings:
      # ... existing port mappings
    kubeadmConfigPatches:
      - |
        kind: ClusterConfiguration
        apiServer:
          extraArgs:
            oidc-issuer-url: http://localhost:8080/realms/confluent
            oidc-client-id: kubernetes
```

### ConfluentRoleBinding Not Applied

**Problem:** User permissions not working in CMF

**Solution:**
```bash
# Verify ConfluentRoleBinding exists
kubectl get confluentrolebinding -n operator

# Check CMF logs for RBAC errors
kubectl logs -n operator -l app.kubernetes.io/name=cmf

# Verify principal name matches Keycloak user email
# In Keycloak: Users → user1 → Email = user1@osow.ski
# In ConfluentRolebinding: spec.principal.name = user1@osow.ski
```

### Keycloak Token Expiration

**Problem:** Users getting 401 Unauthorized after 5 minutes

**Solution:**

Increase access token lifespan in Keycloak:
1. Keycloak Admin Console → Realm Settings → Tokens
2. Access Token Lifespan: Change from 5 minutes to 60 minutes
3. Click **Save**

Or configure token refresh in Control Center:
```yaml
spec:
  configOverrides:
    server:
      - confluent.controlcenter.auth.session.timeout.ms=3600000
      - confluent.controlcenter.auth.oidc.refresh.enabled=true
```
